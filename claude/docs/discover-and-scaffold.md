# /discover と /scaffold — 探索型コマンド

## 経緯

### 問題

Claude Code の継続稼働を研究する中で、以下の構造的な問題が判明した:

```text
レイヤー 3: タスク実行 ← /solve-issue で自動化済み
レイヤー 2: タスク定義 ← Issue テンプレートで標準化できる
レイヤー 1: 方向性の決定 ← ここが曖昧だと全てがブレる
```

`/solve-issue` は Issue を解決するのはスムーズだが、Issue を起票するまでに
「どういう問題があるか」「どう設計するか」を詳細に書く必要がある。
この「言語化コスト」が人間側の最大のボトルネックだった。

### 具体的な課題

- **0 からサービスを作るとき**: 技術スタック、API 設計、DB 設計、インフラ構成など、
  言語化すべき判断が多すぎて Issue に落とすまでが遠い
- **UI/UX に問題があるとき**:「なんか使いにくい」は感じるが、具体的にどのコンポーネントがどう悪いかを文字にするのが手間

### 解決策

Issue を「書く」のではなく「選ぶ」に変える。
人間の負荷を「作文」から「意思決定」にシフトする。

## コマンド体系

```text
/scaffold "こういうサービスを作りたい"
   → 技術設計書 + Issue 一覧を生成
   → 人間: 承認 or 修正
   → /solve-issue #1, #2, #3... を順次実行

/discover "UI がいまいち"
   → コード分析 + 問題点リスト
   → 人間: やるものを選ぶ
   → /solve-issue で実行
```

### 全体の流れ

| フェーズ | コマンド        | 人間の役割         | Claude の役割            |
| :------- | :-------------- | :----------------- | :----------------------- |
| 構想     | `/scaffold`     | 方向性の提示・承認 | 設計の叩き台・Issue 分解 |
| 発見     | `/discover`     | 気づき・選択       | コード分析・Issue 草案   |
| 計画     | `/create-issue` | 確認               | Issue 起票               |
| 実行     | `/solve-issue`  | レビュー           | 実装・PR 作成            |
| 完了     | `/close-issue`  | 動作確認           | ブランチ片付け           |

## /discover

### 用途

既存のコードベースに対して、曖昧な課題意識から具体的な Issue を生成する。

### 入力例

```text
/discover "フロントのデザインがダサい"
/discover "パフォーマンスが遅い気がする" --repo cyg-ai-tech/spark-conversation-api
/discover "セキュリティ面で不安がある"
/discover "UI/UX を改善したい" --screenshot ~/screenshots/current-ui.png
```

### 分析カテゴリ

| カテゴリ       | 分析の重点                                         |
| :------------- | :------------------------------------------------- |
| UI/UX          | コンポーネント構造、レスポンシブ、アクセシビリティ |
| パフォーマンス | N+1 クエリ、キャッシュ、バンドルサイズ             |
| セキュリティ   | OWASP Top 10、認証認可、入力バリデーション         |
| コード品質     | 重複、複雑度、テストカバレッジ                     |
| アーキテクチャ | 結合度、責務分離、スケーラビリティ                 |

### 出力

- 問題点リスト（深刻度・工数・影響範囲付き）
- 選択した問題の Issue 草案
- 承認後に Issue を起票

## /scaffold

### 用途

新規サービスや大きな機能追加の設計叩き台と実装計画を生成する。

### 入力例

```text
/scaffold "特許文書を検索・要約できるチャットボット"
/scaffold "既存の検索をベクトル検索に置き換える" --repo cyg-ai-tech/spark-patent-search
/scaffold "管理画面を作りたい" --reference cyg-ai-tech/spark-frontend-v2
```

### 出力

- ADR 風の設計ドキュメント（技術スタック・アーキテクチャ・要判断事項）
- フェーズ分割された実装計画
- 依存関係付きの Issue セット
- 承認後に Issue を一括起票

### 特徴

- Organization 内の既存リポジトリを分析して、チームのパターンに沿った提案をする
- 不明点は仮置きで進め「要判断事項」として明示する（質問攻めにしない）
- 設計ドキュメントのレビュー→承認を経てから Issue 化する（手戻り防止）

## 活用のコツ

### 入力は雑でよい

「なんか遅い」「デザインがダサい」「こういうの作りたい」で十分。
Claude が足りない情報を自分で調べるか、最小限の質問で補う。

### 選ぶだけでよい

分析結果や Issue 草案は Claude が生成する。
人間は「これはやる、これはやらない」を選ぶだけ。

### /solve-issue と組み合わせる

生成された Issue は `/solve-issue` でそのまま実行可能な粒度になっている。
`/discover` → `/solve-issue` または `/scaffold` → `/solve-issue` の流れで
方向性の決定から実装完了まで一気通貫で進められる。
